Architecture Notes

Architecture role: Streaming Pipeline operates within the Data Pipeline Projects layer of your Azure data platform. Typical flow: source systems → ingestion (ADF/Event Hubs) → bronze landing (ADLS) → silver/gold transformation (Databricks/Synapse) → serving (SQL Warehouse, Power BI).
Component interaction: Key resources include Ingestion, Transformation, Quality, Orchestration, Monitoring, Serving. Each component has a specific responsibility — avoid mixing orchestration logic with transformation logic; keep notebooks idempotent and pipelines metadata-driven where possible.
Data flow: For streaming pipelines, data moves through defined zones with clear ownership. Use managed identities for service-to-service auth, store secrets in Azure Key Vault, and enforce network isolation with private endpoints on production workloads.
Operational model: Deploy infrastructure with Terraform or Bicep, store pipeline code in Git (Azure DevOps/GitHub), promote through dev → test → prod environments, and monitor with Azure Monitor alerts on failure rates and data latency SLAs.

Detailed Notes

What it is: Real-time ingestion with Event Hubs and Spark Streaming. In Azure implementations, Streaming Pipeline typically involves configuring Ingestion and Transformation with clear separation between control plane (orchestration) and data plane (storage/compute).

Platform context: Production pipelines require non-functional requirements: RPO/RTO targets, data freshness SLAs, cost budgets, and compliance constraints (GDPR, HIPAA, SOX). Design for observability first — every pipeline run should emit structured logs with correlation IDs traceable in Log Analytics KQL queries.

How it works: When a pipeline or job runs, the Data Pipeline Projects control plane authenticates via managed identity, reads configuration from Git or a metadata store, executes activities against the data plane, and writes results to ADLS or a warehouse. Checkpointing and logging ensure recoverability.

When to use: Choose streaming pipeline when your workload requires real-time ingestion, watermarks, and exactly-once semantics. Avoid it when a simpler pattern (e.g., direct copy without transformation) meets the SLA at lower cost.

Azure-specific configuration: Provision resources in the same region as your data lake to minimize egress charges. Use Azure Policy to enforce tags, encryption, and allowed SKUs. Register resources under End-to-end Azure data platform in your subscription before deployment.

Streaming specifics: Use Event Hubs or Kafka as the buffer, Structured Streaming with checkpoint locations on ADLS, and Delta Lake as the sink for exactly-once semantics. Set watermarks for late-arriving data and monitor consumer lag.

Production Notes

Best Practices

  • Use managed identities instead of embedded credentials for all Data Pipeline Projects connections
  • Parameterize pipelines and notebooks — avoid hard-coded environment-specific paths
  • Implement idempotent writes (MERGE/upsert) so retries do not duplicate data
  • Partition data by date or high-cardinality filter columns to optimize query pruning
  • Set up diagnostic settings to Log Analytics on day one — not after the first incident
  • Document data contracts: schema, freshness SLA, owner, and downstream consumers for every table
  • Test with representative data volumes in a staging environment before production cutover
  • Apply the principle of least privilege — separate dev/test/prod service principals and RBAC scopes

Common Pitfalls

  • Running production workloads on public endpoints without private link or firewall rules
  • Ignoring small-file problems — millions of tiny Parquet files destroy Spark read performance
  • Skipping data quality checks at the silver layer and pushing dirty data to gold dashboards
  • Hard-coding connection strings or storage paths instead of using Key Vault and parameters
  • Not setting retry policies and alerts — silent failures propagate stale data to executives
  • Over-partitioning or under-partitioning — both cause shuffle-heavy slow queries
  • Deploying schema changes without backward-compatible evolution or consumer notification

Interview Preparation

  • Be ready to whiteboard how streaming pipeline fits into a Data Pipeline Projects architecture with ADLS, ADF, and Databricks
  • Explain trade-offs: cost vs performance, batch vs streaming, schema-on-read vs schema-on-write
  • Describe a real incident: what failed, how you diagnosed it (Monitor/Spark UI/ADF runs), and how you fixed it
  • Discuss streaming pipelines — interviewers want specifics, not generic definitions
  • Mention governance: Unity Catalog, column-level security, and PII masking for regulated industries

Code Example

# Streaming Pipeline — PySpark / Delta Lake example
from pyspark.sql import functions as F

# Read source (ADLS Gen2 / Delta)
df = spark.read.format("delta").load("abfss://bronze@storage.dfs.core.windows.net/streaming-pipeline/")

# Transform — Real-time ingestion with Event Hubs and Spark Streaming
result = (
    df.filter(F.col("is_active") == True)
      .withColumn("processed_at", F.current_timestamp())
      .dropDuplicates(["id"])
)

# Write to silver/gold Delta table (idempotent MERGE pattern)
result.write.format("delta").mode("overwrite").save(
    "abfss://silver@storage.dfs.core.windows.net/streaming-pipeline/"
)
