1. What Is Apache Spark?

Apache Spark is an open-source, distributed computing engine designed to process large-scale data (terabytes and beyond) across many machines in parallel. Instead of one computer loading 30 GB into RAM, Spark divides the data into partitions, sends each partition to a different CPU core, and runs transformations simultaneously — finishing in minutes instead of hours.

⚡ In-Memory Processing

Keeps intermediate results in RAM (not disk), making iterative jobs 10–100× faster than MapReduce.

📊 Unified Engine

Supports SQL (Spark SQL), streaming, ML (MLlib), and graph processing (GraphX) in one framework.

🔄 Lazy Evaluation

Transformations build a plan (DAG) first; nothing runs until an action triggers execution.

🛡️ Fault Tolerance

Tracks lineage — if a partition is lost, Spark recomputes it from the original source automatically.

🧠 Driver

Runs your main() function. Converts your code into a DAG of stages, splits stages into tasks, and sends tasks to executors. Collects results.

📋 Cluster Manager

Negotiates resources. Tells Spark how many executors and how much memory each gets. Does NOT run your code.

🖥️ Worker Node

A physical or virtual machine in the cluster. Hosts one or more executor JVM processes.

⚙️ Executor

A JVM process on a worker. Runs tasks, stores cached data in memory/disk, and reports back to the driver.

4. Driver Internals — What Happens Inside the Driver JVM

The Driver is not just "your script" — it runs a stack of schedulers and managers that convert SQL/DataFrame code into distributed work.

🔌 SparkContext

Entry point to Spark. Holds connection to Cluster Manager, creates RDDs, registers the app. One per JVM. Created when SparkSession.builder().getOrCreate() runs.

📐 DAGScheduler

Splits the logical DAG into stages at shuffle boundaries. Submits stage task sets to TaskScheduler. Tracks stage completion and handles stage retries on failure.

⏱️ TaskScheduler

Assigns tasks to executors based on data locality (PROCESS_LOCAL preferred). Maintains task queue, handles task retries, enables speculative execution for stragglers.

📦 BlockManager (Driver)

Tracks where shuffle blocks and cached RDD blocks live across executors. Driver BlockManager coordinates metadata; executor BlockManagers store/fetch actual bytes.

Driver lifecycle for our 30 GB job (step-by-step)
  1. T+0ms: SparkSession creates SparkContext — registers app "30GB-Sales" with YARN/K8s.
  2. T+200ms: Cluster Manager allocates 4 executors (4 cores × 4 GB each). Driver receives executor registration RPCs.
  3. T+500ms: User code builds lazy DataFrame chain — Catalyst parses SQL, no tasks yet.
  4. T+800ms: write.parquet() action → DAGScheduler creates Job 0 with 3 stages.
  5. T+850ms: DAGScheduler submits Stage 1 (240 tasks) to TaskScheduler.
  6. T+850ms–4min: TaskScheduler launches tasks in batches of 16, tracks completions via RPC.
  7. T+4min: Stage 1 done → shuffle write complete → Stage 2 submitted.
  8. T+6min: Stage 2 + write complete → Driver marks Job 0 SUCCEEDED.

How HDFS replication affects Spark reads

HDFS stores each 128 MB block with replication factor 3 (default). Block locations are reported to Spark's FileScan operator.

  • TaskScheduler waits up to spark.locality.wait (default 3s) for a PROCESS_LOCAL slot before launching NODE_LOCAL.
  • Each executor reads its partition via HDFS client — often local disk if block replica is on same node (~125 MB/s SSD).
  • Network read only when no local replica: ~1 Gbps cluster network → ~19 GB filtered shuffle is the real bottleneck, not initial read.
Component: TaskScheduler Memory: HDFS client buffer ~64 KB batches Disk: 128 MB block read per task

7. Tungsten Execution Engine

Spark's physical engine uses binary in-memory formats, whole-stage codegen, and optional off-heap memory for speed.

⚙️ Whole-Stage Codegen

Stage 1 fuses read+filter into one generated Java function — no virtual function calls per row. ~2× faster than interpreted mode.

📊 Binary Internal Format

Rows stored as compact bytes (UnsafeRow) — no Java object overhead. 128 MB partition ≈ millions of rows in ~79 MB after filter in execution memory.

🧠 Off-Heap Memory

spark.memory.offHeap.enabled=true uses direct buffers outside JVM heap — reduces GC pauses on large shuffles.

💾 Spill to Disk

When execution memory exceeds limit, Tungsten sorts spill to local disk (/tmp/spark-*). Job continues — Spark UI shows "Shuffle Spill (Disk)".

spark.sql.codegen.wholeStagetrueFused Stage 1 pipeline

8. Shuffle Mechanics — Map Side, Sort, Spill, Fetch

The groupBy("region") shuffle is the most expensive phase — ~19 GB filtered data redistributed across 200 partitions.

Map Side (Stage 1 tasks)
  1. Each row hashed by region → partition ID (0–199)
  2. Map-side combine: partial sums optional
  3. In-memory sort per partition bucket
  4. Spill to disk if memory full (~shuffle write files)
  5. Merge spill files → final shuffle data file
Disk: ~19 GB write Memory: 32 MB shuffle buffer/task
⇄
network
Reduce Side (Stage 2 tasks)
  1. BlockManager fetches remote shuffle blocks
  2. External shuffle service serves disk files
  3. Merge fetched data in memory
  4. HashAggregate: sum(amount) per region
  5. Output: ~50 regions × 8 bytes ≈ 400 bytes/task
Network: ~19 GB fetch total Memory: aggregate hash map

9. Broadcast Variables & Speculative Execution

📡 Broadcast (not used here, but relevant)

If we joined with a small lookup table (<10 MB), Spark would broadcast() it to all executors via BitTorrent-style transfer — one copy per executor in Storage Memory, no shuffle for the small side.

🏃 Speculative Execution

When spark.speculation=true, TaskScheduler detects stragglers (task > median × 1.5). Launches duplicate task on another executor. First to finish wins — critical when 1 of 240 tasks is slow.

13. Memory Management Inside an Executor

Each executor JVM divides its allocated memory carefully so compute and cached data coexist safely.

Unified Memory Model (Spark 1.6+)

Storage Memory (~50%)
Cached RDDs/DataFrames, broadcast variables, shuffle data buffers
Execution Memory (~50%)
Sorts, joins, aggregations, temporary data structures during tasks

If memory is full, Spark spills to disk — the job continues but slower. For 30 GB input with 4 GB executors, most data is read partition-by-partition (not all cached), so spill is minimal if partitions are ~128 MB. Stage 1 shuffle write may spill ~890 MB to disk across cluster.
